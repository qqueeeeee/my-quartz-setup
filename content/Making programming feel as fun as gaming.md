# Making programming feel as fun as gaming

This is me trying to pin down what kind of programming work can actually hit the same reward loops that gaming does.

## Notes

I wanted to dedicate serious time to finding my programming niche, with the explicit goal of replacing gaming with programming by making programming feel just as engaging. The core challenge was pretty clear: gaming has tight feedback loops, visible progression, mastery, and stakes, and I wanted to find a corner of programming that hits the same way. Claude laid out four directions that actually fit me: competitive programming, game development, systems/performance work, and security/CTF stuff. I already have a background in reverse engineering, building a private server from packet captures, and doing way too much desktop/tooling tweaking, so the strongest fit seemed to be systems or CTF work because it matches that same hacker curiosity. The deeper issue, though, was not just picking a niche. It was that I have too many ideas, freeze when there are too many options, and then default back to gaming because it has almost zero activation energy and a guaranteed reward loop. Claude reframed that less as a motivation problem and more as an activation-energy problem. The conversation ended on the one question that actually matters: what is the last thing I saw that made me genuinely excited enough to chase it properly?
